Binary Ninja Introduction

Posted on Wed 21 August 2019 in Research

Introduction

Recently, I purchased a personal license for Binary Ninja from Vector 35 LLC. In this post I plan on covering some of the reasons I took the plunge and some of the features of Binary Ninja.

Binary Ninja is a binary analysis platform heavily focused on static analysis. With plans for the future of adding features like dynamic analysis, decompilation, and collaboration. Note: some users have already made plugins allowing dynamic analysis possible.

Before diving too deep into this blog post I just want to point out I am making an attempt to avoid comparisons to other products that fall into the same general product space. I will leave it up to the reader to make decisions based on their own requirements for tooling and intentions. I am simply writing about my experiences, and my reasoning for going with Binary Ninja as it applies to my use cases (Vulnerability Research and Exploit Development). It is totally possible that your use case will have a lot more specific requirements than mine. If that is the case Binary Ninja might not be right for you, but I urge you to at least give it a try and look into it. These are all my opinions and the reader is encouraged to form their own opinions.

Positives

For me the decision to purchase Binary Ninja boiled down to a few different reasons. Firstly, the price point. At the time of writing this post a personal license costs $149 USD. This is a pretty reasonable price considering all that Binary Ninja has to offer.

It is worth noting that for the price point you also get a version of Binary Ninja for OSX, Linux, and Windows. This is very convenient for someone who uses all three operating systems on a regular basis.

Secondly, the licensing model fits my needs very nicely. I plan on bug hunting in an effort to sell my research to bug bounty programs. The licensing allows me to purchase a personal license and eventually upgrade to commercial when I start making enough money from this endeavor to afford one on a regular basis. More about this can be found on the Binary Ninja FAQ page [1].

Another big selling point for me was the API. The API provides a very powerful capabilities for plugin authors. Also, just looking at the API documentation [2] it is clear a lot of time and thought went into the design and development of the API. Everything is well organized, clearly named, well documented, and in a lot of cases the documentation provides usage examples. You can even look at the source code for much of the API.

The combination of a powerful API and the innovative Binary Ninja Intermediate Languages (BNIL) is yet again another huge selling point in my opinion. Binary Ninja translates assembly code to an initial Intermediate Language (IL) that can then be used to translate to other IL forms representing different layers of abstraction from the original assembly code. BNIL is well designed and easy to lift in comparison to other IL languages that are very verbose, complex, and hard to understand.

Negatives

I want to be as transparent as possible here. I enjoy using Binary Ninja but I also realize it might not be the perfect tool for everybody. It has a lot of features that fit my use cases which are vulnerability research, exploit development and CTFs. In other words, your milage may vary. I will attempt to list some of the things I have found lacking so far. This might not be all encompassing so do your research if you have a specific use case in mind.

While Binary Ninja currently falls short in some areas, for example function matching, full PDB support (It does do a decent job but is lacking some aspects of support for PDB symbols), and limited architecture support (currently it supports roughly 11 CPU architectures, some via plugins). Lack of floating point operation support for BNIL. If any of these issues are a deal breaker for you, I suggest checking back regularly as development is moving quickly. Also, keep in mind there are plenty of things Binary Ninja offers other tools simply do not.

Early on it was pretty clear that the developers of Binary Ninja were CTF players. A lot of the features are geared towards exploitation and CTF like challenges. If your particular use case does not fall in this category it might be hard to fit Binary Ninja into your normal work flow. They have been slowly expanding the capabilities to fit more reversing tasks and workflows. For example: Binary Ninja might not be the best tool for analyzing complex malware. Vector35 has even stated they have not fully tested Binary Ninja against potential adversarial binaries.

The Vector35 team have proven to me in a very short time, based on my interactions with them on the support slack channel, that they are extremely dedicated to consistently improving the product. Additionally, the community has been great. The community is very active and always producing cool plugins and features. And as I mentioned above, the price point plus the powerful API allows capable users to overcome a vast majority of these short comings if they truly want to spend the time to do so.

Features

Now that I have covered some of the selling points of Binary Ninja that I found attractive, and some of the negative aspects, lets dig into some of the features a bit.

API

BNIL

BNIL is a pretty innovative approach to IL. Rather than create a single IL language, BNIL is comprised of multiple IL abstraction layers Low Level Intermediate Langauge (LLIL) and Medium Level Intermediate Language (MLIL) with High Level Intermediate Language (HLIL) still in development. This means that there are different levels of abstraction and the user can pick and choose the one that best suites their needs for any given purpose.

Binary Ninja Intermediate Language Abstraction Flow

Additionally, Binary Ninja has introduced Single Static Assignment (SSA) forms for each IL. This is useful for various applications such as emulation, SMT solving, and Taint Analysis.

For me BNIL was a hugely attractive, as I had read a few blog posts on modeling vulnerabilities [4] [7] in Binary Ninja. I had always had the idea writing a library of vulnerability detections for binary code. And from the looks of it BNIL seems like a good place to start.

Customization

It is obviously apparent a lot of thought has gone into making things as customizable as possible. From keybindings to themes, Binary Ninja has you covered.

Currently there are roughly 450 actions you can create custom keybindings for in Binary Ninja. Jordan Wiens (Vector35) went so far as to create custom keybindings that mimic VIM motions [6].

I know on the surface it might seem superficial, but having a comfortable working environment goes a long way. For me themes are a big deal, if I am spending a lot of time staring at a computer screen I much prefer looking at something that is easy on the eyes.

Binary Ninja happens to come with a nice default dark theme. But Binary Ninja also has a ton of community created themes [[5]], including the common base16 themes VIM users might expect.

Binary Ninja Default Theme

Conclusion

Binary Ninja is a great product. It has a ton of nice features and offers the power to customize and extend almost everything about it.

While Binary Ninja might not be for everybody, I think it is an up and coming product this market space has needed for some time now.
